# 基于 RAG 的多模态电商智能导购 AI Agent — 开发计划

> 技术栈：**Android (Kotlin) + FastAPI (Python) + Chroma 向量库**
> 目标：**先跑通最小闭环 → 再做多轮对话与反选 → 全程为加分项预留接口**
> 适用：1–2 周、单人开发

---

## 0. 总体策略

这份计划遵循 PDF 里反复强调的核心原则——**"先跑通最小闭环，再逐步增强；评审重点关注端到端链路的完整性"**。所以排期不是按"功能清单"铺开，而是按"**每个阶段结束都有一个能演示的可运行版本**"来切分。任何一个阶段被迫中断，上一阶段的产物都仍然是一个完整、可答辩的 Demo。

加分项不在 1–2 周单人范围内强行全做。计划的做法是：**在架构层面为四类加分项（购物车下单 CRUD、多模态、对比决策、性能优化）全部预留接口和适配层**，但只在主线里深度实现"多轮对话 + 反选"。这样既符合评审"重深度而非广度"的导向，又保证后续想补任何一项时，改动都集中在一个新增模块里、不动主链路。

评分权重决定了精力分配：基础功能完整性 35%、工程质量 25%、效果与可靠性 20%、加分项 20%。因此**最小闭环和工程规范（占 60%）必须做扎实**，加分项做一项有深度即可。

## 1. 技术选型与理由

| 层次 | 选型 | 说明 |
|---|---|---|
| 客户端 | Android + Kotlin，Jetpack Compose | 原生开发（PDF 硬性要求，纯 Web/H5 会扣分）。Compose 做逐字流式渲染和商品卡片比 XML 布局更省事 |
| 网络层 | OkHttp + SSE（`okhttp-sse`） | 最小闭环用 SSE 足够；若后续做语音双向流再升级 WebSocket |
| 后端框架 | FastAPI | 原生 `async`，`StreamingResponse` 直接支持 SSE，Pydantic 做请求/响应模型校验 |
| 向量库 | **Chroma**（本地持久化模式） | 50–100 条数据量下，Chroma 零运维、`pip install` 即用、自带持久化，最适合单人小项目。架构上通过 `VectorStore` 抽象接口隔离，后期想换 Qdrant/Milvus 只改一个适配类 |
| Embedding | Doubao-embedding-vision | PDF 推荐；选 vision 版是为给"拍照找货"加分项留多模态向量空间。先用文本能力，后期直接接图像 |
| 大模型 | Doubao-Seed-2.0-lite | PDF 已提供 API Key，OpenAI 兼容接口 |
| 配置管理 | `.env` + `pydantic-settings` | **API Key 必须放 `.env` 并写入 `.gitignore`**（PDF 明确警告） |

**为什么是 Chroma 而不是其他：** 数据量极小、单人开发、要求快速跑通——Chroma 在这个场景下综合成本最低。Milvus/Qdrant 适合生产规模，但部署和运维开销在这里是负担。关键是**架构上不把 Chroma 写死**：所有检索调用都走 `VectorStore` 抽象接口，换库是局部替换而非重构。

## 2. 架构设计（含接口预留）

### 2.1 端到端链路

```
Android App
  │  对话输入（文字 / 预留：语音、图片）
  ▼
[SSE/WebSocket]  ──►  FastAPI 网关层
                        │
                        ▼
                   Agent 编排层（Orchestrator）
                   ├─ 意图识别（闲逛 vs 明确购买）
                   ├─ 查询改写（多轮上下文 / 反选条件解析）
                   ├─ 检索（VectorStore 抽象接口 → Chroma）
                   ├─ 后处理（结构化过滤、排除、范围筛选）
                   └─ 生成（LLM 流式输出 + 商品卡片结构化数据）
                        │
                        ▼
                   流式回包（文本增量 + product_card 事件）
  ▼
Android 逐字渲染 + 可点击商品卡片
```

### 2.2 关键抽象（这是"flexible"的核心）

计划在后端定义五个抽象接口/扩展点，让加分项变成"插模块"而不是"改链路"：

1. **`VectorStore` 接口** — `add()` / `query()` / `delete()`。当前实现 `ChromaStore`，预留 `QdrantStore`。
2. **`ToolRegistry` 工具注册表** — Agent 编排层通过工具调用执行动作。最小闭环只注册 `search_products` 一个工具；购物车加分项就是新增 `add_to_cart` / `update_cart` / `checkout` 工具，**编排逻辑不变**。
3. **`InputProcessor` 输入处理链** — 当前只有 `TextProcessor`。多模态加分项是新增 `ASRProcessor`（语音转文字）、`VLMProcessor`（图片转检索语义），它们的产物统一归一成"文本查询 + 可选向量"，再走同一条 RAG 链路。
4. **`PostProcessor` 检索后处理器** — 责任链模式。最小闭环放 `RangeFilter`（价格区间）；反选加分项新增 `ExclusionFilter`（排除属性/品牌）；对比加分项新增 `ComparisonAggregator`。
5. **流式事件协议** — 回包用统一的事件类型（`token` / `product_card` / `cart_update` / `action` / `done`）。客户端按事件类型分发渲染。新增加分项只是新增事件类型，老客户端忽略未知类型即可向后兼容。

> **设计意图：** 任何一个加分项的实现，理想情况下只新增一个 `Tool`、一个 `Processor`、或一个事件类型，**不修改 Orchestrator 主流程**。这就是"为未来留好接口和适配"的具体落点。

### 2.3 会话状态设计（为多轮预留）

最小闭环阶段就引入 `SessionState`（按 `session_id` 存内存字典即可，预留 Redis 适配），字段包括：对话历史、已确认的筛选条件（含正向条件和**排除条件**两个列表）、当前候选商品集、购物车占位字段。最小闭环阶段这些字段大多为空，但**结构先立起来**，多轮和反选阶段直接往里填，不动数据模型。

## 3. 分阶段开发计划

四个阶段，每阶段结束都有可演示版本。下方"产出"即该阶段的验收物。

### 阶段 0 — 准备与脚手架（约 1 天）

| 任务 | 说明 |
|---|---|
| 仓库初始化 | 按 PDF 要求分目录：`/client`、`/server`、`/docs`；配 `.gitignore`（务必含 `.env`） |
| 数据预处理 | 解析 50–100 条电商数据（商品名/类目/价格/详情/主图 URL），清洗成统一 JSON |
| Chunking 策略 | 每个商品按"基础信息块 + 详情描述块"切分；价格、库存等结构化字段**单独存为 metadata**，不进向量、用于精确过滤 |
| 灌库脚本 | `ingest.py`：调用 Embedding 写入 Chroma，持久化到本地目录 |
| 环境联通 | FastAPI 起一个 `/health`；Android 起一个空壳 App 能发请求 |

**产出：** 向量库可检索（命令行验证）、前后端能握手。

> Chunking 建议：商品文案短，过度切分会丢上下文。粒度定在"一商品一主块 + 一详情块"即可，后续若召回不准再调。

### 阶段 1 — 最小闭环 MVP（约 3–4 天）★ 重点

对应 PDF "3.5 基础功能要求"和"最小功能交付标准"。这是 35% 权重所在，必须扎实。

| 模块 | 任务 |
|---|---|
| 后端 - RAG 链路 | 实现 `search_products`：查询 → Embedding → Chroma 检索 Top-K → 组装上下文 |
| 后端 - 生成 | 系统提示词设计：**严格约束只能用检索结果，禁止编造商品/价格/优惠**（PDF 减分项第一条）；输出含推荐理由 |
| 后端 - 流式 API | `POST /chat` 用 SSE 流式返回 `token` 增量；商品以 `product_card` 事件结构化下发 |
| 后端 - 工程 | `ToolRegistry` 注册一个工具；`VectorStore` 抽象落地；统一异常处理（超时、检索空结果降级话术） |
| 客户端 - 对话 | Compose 对话界面，发文字、接 SSE、**逐字流式渲染**（仿"豆包"打字机效果） |
| 客户端 - 卡片 | `product_card` 事件渲染成可点击商品卡片，点击跳详情/落地页占位 |

**产出（可答辩 Demo）：** 用户输入"推荐一款适合油皮的洗面奶"→ 流式回复 + 商品卡片。已满足 PDF "合格标准"全部 5 条。

**自测重点：** 幻觉检查（问库里没有的商品，看是否老实说"没有"）、检索空结果、网络中断、长响应。

### 阶段 2 — 多轮对话 + 反选（约 3–4 天）★ 重点加分项

对应 PDF "4.3 对话智能与 RAG 增强"的 ⭐ 多轮上下文 与 ⭐⭐ 反选排除。

| 模块 | 任务 |
|---|---|
| 多轮上下文 | 启用 `SessionState`：每轮把历史 + 已确认条件喂给 LLM；做**查询改写**——把"再便宜点的呢？"补全为"预算更低的 X 类商品" |
| 渐进收敛 | 用户追加条件（"要轻量的"→"预算 500 以内"）时，条件累加进 `SessionState.filters`，候选集逐步收窄 |
| 反选解析 | LLM 把"不要含酒精的""除了耐克"解析成结构化排除条件，写入 `SessionState.exclusions` |
| 反选执行 | 新增 `ExclusionFilter` 后处理器：在检索结果上按 metadata / 关键词排除特定属性、品牌 |
| 意图识别 | 区分"随便看看"与"明确购买倾向"，触发不同检索/话术策略（PDF 3.2） |
| 客户端 | 多轮 UI 衔接、上下文连贯展示；条件可视化（可选：把已选条件做成标签 chip） |

**产出（可答辩 Demo）：** 完整多轮链路 ——"帮我推荐跑鞋"→"要轻量的"→"预算 500 以内"→"不要日系品牌"，候选逐步精准。已达到 PDF "优秀标准"的"支持多轮对话和反选等复杂场景"。

**这是计划里深度实现的加分方向**，对应评审"做精一项胜过浅尝三项"。

### 阶段 3 — 接口预留固化 + 打磨（约 2–3 天）

不深做其余加分项，但要把**预留接口写成可演示的骨架**，并打磨工程质量与体验（占 25% + 20%）。

| 方向 | 本阶段做到的程度（预留，非完整实现） |
|---|---|
| 购物车与下单 CRUD | 后端 `SessionState` 已含购物车字段；定义 `add_to_cart` / `update_cart` / `checkout` 工具的**空实现 + 接口签名**；流式协议预留 `cart_update` 事件。客户端预留购物车状态区。→ 后续补全只需填工具逻辑 |
| 多模态 | `InputProcessor` 链已抽象；预留 `ASRProcessor` / `VLMProcessor` 接口和路由分支。客户端预留语音按钮、拍照入口（占位）。→ 后续接入只需实现 processor |
| 多商品对比 | 预留 `ComparisonAggregator` 后处理器接口和 `comparison_card` 事件类型 |
| 性能优化 | 实现**热门查询缓存**（轻量、性价比最高的一项）：对高频相同查询缓存结果，降低延迟和模型调用。预留首屏优化、骨架屏的位置 |
| 工程质量 | 完善 README、`requirements.txt` 版本锁定、核心逻辑（RAG 链路、Prompt 构造）注释；错误处理补全 |
| 体验打磨 | 流畅动效、骨架屏加载、卡片交互细节 |

**产出：** 一个端到端流畅、工程规范齐全的 Demo，外加四类加分项的清晰扩展点（答辩时可讲"如何扩展"）。

## 6. 验收自检清单（对照评分维度）

**基础功能完整性（35%）**
- [ ] 端到端链路跑通：对话 → RAG 检索 → 模型生成 → 流式返回 → 商品卡片
- [ ] 单轮模糊推荐可用，流式输出正常
- [ ] 商品卡片可展示、可点击

**工程质量（25%）**
- [ ] 前后端分目录，结构清晰
- [ ] 依赖项与版本号明确（requirements.txt / Gradle）
- [ ] 核心逻辑（RAG 链路、Prompt 构造）有注释
- [ ] 错误处理完善（超时、并发、空结果降级）
- [ ] README + 技术文档齐全

**效果与可靠性（20%）**
- [ ] 无幻觉：不编造商品、价格、优惠
- [ ] 多轮上下文连贯，反选语义正确
- [ ] 运行流畅、界面美观、无明显 Bug

**加分项深度（20%）**
- [ ] 多轮 + 反选深度实现（主线）
- [ ] 热门查询缓存（性能项）
- [ ] 购物车/多模态/对比的扩展接口清晰，可在答辩中讲明扩展方式

**减分项规避**
- [ ] API Key 在 `.env` 且已 `.gitignore`
- [ ] 原生 App，非 Web/H5
- [ ] Demo 开箱即跑，无需大量手动配置
- [ ] 能解释代码原理（避免纯 AI 生成不可解释）

## 7. 主要风险与应对

| 风险 | 应对 |
|---|---|
| 数据量小（50–100 条）导致召回不准 | 精心设计数据覆盖多类目；调 Chunking 粒度；必要时按 PDF Q2 适当扩充 Case |
| 流式渲染在 Android 端卡顿 | 优先用 Compose；SSE 增量按字/词刷新，避免整段重绘 |
| 多轮上下文越长越易跑偏 | 用查询改写压缩历史；`SessionState` 只保留结构化条件而非全量历史 |
| 反选语义被模型忽略 | 不靠 Prompt 兜底，用 `ExclusionFilter` 在检索结果上**程序化强制过滤** |
| 单人工期紧 | 严守"阶段优先级"：最小闭环 > 多轮反选 > 加分项预留，任一阶段超时则砍后续而非砍质量 |

---

*本计划以 PDF《课题说明会：基于 RAG 的多模态电商智能导购 AI Agent》为依据编写。核心思想：最小闭环做扎实拿基础分，多轮+反选做深拿加分，其余加分项以接口预留保证未来可低成本扩展。*
